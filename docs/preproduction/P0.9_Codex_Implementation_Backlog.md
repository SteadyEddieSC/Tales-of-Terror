# P0.9 Codex Implementation Backlog

**Status:** Draft issue backlog only
**Implementation authorization:** None
**Required starting point:** Clean local `main` reconciled to protected `origin/main` after P0.9

These issues are intentionally small and ordered. Codex must report repository state and wait for explicit authorization before starting any issue.

## 1. Local repository reconciliation

**Goal:** Reconcile the local Windows repository to protected main without discarding unknown work.

**Allowed actions:**

- inspect status, branches, commits, remotes, and tool versions;
- fetch and fast-forward a clean local `main`;
- run all repository and P0.x validators;
- report issue #44 audit state.

**Excluded:**

- feature development;
- destructive reset or clean;
- branch deletion without review;
- dependency changes;
- Drowned Harbor runtime work.

**Acceptance:** Complete the report required by `P0.8_Codex_Return_Handoff_v1.md`.

## 2. Companion dependency remediation — issue #44

**Goal:** Remove or upgrade the vulnerable `sharp` dependency chain without suppressing audit findings.

**Likely scope:** Companion package manifests, lockfiles, tests, and GitHub Actions only.

**Required evidence:**

- dependency-tree explanation before and after;
- clean install;
- service tests;
- browser tests;
- dependency audit;
- no protocol or security-boundary regression.

**Excluded:** Drowned Harbor content and Tale authoring implementation.

## 3. Generic authoring-reference inspection CLI

**Goal:** Add a read-only CLI that summarizes any valid `tale_authoring_reference` without compiling or loading it at runtime.

**Suggested files:**

- `tools/tale_authoring_reference.py`;
- `tools/test_tale_authoring_reference.py`;
- documentation and CI only.

**Capabilities:**

- validate;
- print authoring identity;
- list stages, modes, content counts, open blockers, and media dependencies;
- emit deterministic JSON summary;
- never write into `game/data/tales/`.

**Excluded:** Runtime loader, provider, catalog, Godot integration, or production digest changes.

## 4. Board-authority schema design

**Goal:** Define a generic repository-authored board-authority schema compatible with the existing production Tale-package contract.

**Required design coverage:**

- spaces and regions;
- connectors per tide state;
- route type and direction;
- hazard eligibility;
- visibility and blocked state;
- authored recovery;
- deterministic High Water transformation;
- terminal reachability obligations.

**Deliverable:** Documentation, schema, valid and invalid fixtures, validator, tests, and CI.

**Excluded:** Godot board implementation and Drowned Harbor catalog entry.

## 5. Scenario-manifest schema design

**Goal:** Define stages, transitions, triggers, mandatory and optional encounter pools, objective gates, and ending eligibility.

**Required properties:**

- stable IDs;
- deterministic ordering;
- once-only transition identity;
- no private-state-only public transition;
- terminal reachability;
- unsupported mode and seat-count fallback;
- rejected mutation as no-op.

**Excluded:** Runtime scenario executor.

## 6. Rules and social-authority schema design

**Goal:** Define data contracts for resources, items, cards, hazards, roles, factions, forms, transformations, continuation, and outcome attribution.

**Required properties:**

- stable-seat ownership;
- exact privacy classes;
- controlled reveal;
- generic role alternatives;
- hazard warning/response/recovery;
- transformed and Restless agency;
- explicit mixed-outcome attribution.

**Excluded:** Balance values presented as final and runtime rules implementation.

## 7. Authoring-reference consistency reporter

**Goal:** Compare an authoring reference to its P0.2–P0.8 sources and produce a deterministic drift report.

**Checks:**

- missing or renamed stable IDs;
- missing dialogue/media/accessibility references;
- new open blockers;
- privacy mismatch;
- source-anchor drift;
- production-boundary violations.

**Excluded:** Automatic source modification or compilation.

## 8. Shared-screen authoring preview specification

**Goal:** Define a non-game authoring preview for stages, public choices, captions, transcript, control changes, transformations, and endings.

**Deliverable:** P0.10 storyboards and implementation specification before any UI code.

**Excluded:** Production gameplay scenes and claims of television/controller usability.
